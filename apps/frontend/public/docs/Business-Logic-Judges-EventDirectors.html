Business Logic: Judges & Event Directors

How members become Judges or EDs, get certified, get assigned to events, and get rated by competitors after.

Judge / ED Application

Overview
Members submit detailed applications including bio, references, certifications, and acknowledgments.
Technical
judge_applications and event_director_applications tables with 30+ fields plus three references each. Reference rows live in judge_application_references; each reference receives a verification email and can confirm the applicant.
Why
Judging integrity matters. A long form with verifiable references is the lightweight version of a background check. Acknowledgments (independent contractor, code of conduct, terms) are checkboxes on the same form so the legal trail lives with the application.
Manage
/admin/judge-applications and /admin/event-director-applications. Judge & ED Management.

Admin-create paths

Overview
Three additional ways an admin can grant Judge/ED status when a member can't (or shouldn't) fill out the form themselves.
Technical
1) Admin full application — admin enters every field on the member's behalf. 2) Quick-add — minimal fields, placeholders for the rest, auto-approves. 3) Direct judge record — skip the application entirely, create the judges row directly (for grandfathering existing certified judges).
Why
Some long-time judges should never have been required to fill out a form. Some need to be added at the venue on event day. The four paths handle the spectrum from "rigorous" to "expedient."
Manage
See the dedicated Add a Judge or ED Manually deep guide.

Approval & Certification

Overview
Admin reviews application, picks level (Apprentice / Certified / Senior), sets optional expiration.
Technical
Approval creates a judges (or event_directors) row linked to the user, with level, specialty, sub-specialties, certification expiration. Profile gets can_apply_judge = true with grant audit fields populated.
Why
Two states (permission flag on profile + record in judges table) let us distinguish "can apply" (permission) from "currently certified" (record). Splitting allows revoking certification while keeping the historical permission grant.
Manage
Approval modal on the application detail page.

Event Assignment

Overview
Admin (or ED) assigns Judges to an event's judge panel.
Technical
event_judge_assignments rows link a judge to an event with status (assigned / accepted / declined / completed). Judges receive an email notification on assignment.
Why
Status field lets us track invitations, declines, and post-event completion separately. Without it we couldn't spot judges who stopped showing up.
Manage
Event detail page → Judge Panel section.

Level Progression

Overview
Track a Judge moving from Apprentice → Certified → Senior over time.
Technical
judge_level_changes history table — each change records old level, new level, effective date, optional notes. Current level on the judges row is the latest.
Why
Level history matters for promotion fairness ("how long was this person Apprentice before promotion?") and for retro-investigating scoring disputes.
Manage
Judge detail page → Record Level Change.

Ratings After Events

Overview
Members rate the Judges and EDs they competed under; aggregated to spot outliers.
Technical
ratings per (rater, ratee, event) with score (1-5) and comment. The Ratings Analytics page aggregates per-judge and per-ED with date filters.
Why
Direct member feedback is the cleanest signal of judge quality. Aggregating per-event lets us see if a single bad day is dragging an otherwise great judge's average.
Manage
/admin/ratings. Ratings Analytics.
