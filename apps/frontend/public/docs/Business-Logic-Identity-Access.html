Business Logic: Identity & Access

How the platform identifies users, controls who can sign in, and protects against abuse. Every other system on the site sits on top of these primitives.

Authentication (Supabase Auth + JWT)

Overview
Every signed-in user has a Supabase auth account. Sessions are JWT-based, validated by a global guard on every authenticated request.
Technical
Sign-in goes through Supabase Auth which issues an access token (~1 hour TTL) and a refresh token. The access token is sent as a Bearer header on every API call. GlobalAuthGuard validates it via auth.getUser(token) and attaches the user to the request. Public routes opt out via the @Public() decorator.
Why
Outsourcing auth to Supabase means we get hardened password storage, refresh-token rotation, OAuth providers, and email verification for free. JWT-on-every-request is stateless and scales horizontally without a session store.
Manage
No direct admin UI — auth state is managed via member CRUD on /admin/members and the underlying Supabase project console (super-admin only).

Profile records (the application-side user)

Overview
Each Supabase auth user has a paired profiles row that holds everything the platform cares about — name, addresses, role, MECA ID, judge/ED permissions, ban flags.
Technical
Profile id matches the Supabase user id 1:1. Created on first ensure-profile call (post-signup). Holds 50+ fields including role, is_staff, account_type, master_profile (for secondaries), all permission flags, address blocks, billing/shipping, can_login, login_banned, maintenance_login_allowed, and lifecycle timestamps.
Why
Splitting auth from profile lets the dev team migrate auth providers if needed without touching application data. Profile is also where the platform stores everything Supabase doesn't care about (vehicle info, public bio, MECA ID, ban audit fields).
Manage
/admin/members — full search, edit, create, impersonate, delete. See the Member Management guide.

MECA ID Assignment

Overview
Every member gets a unique numeric MECA ID. New IDs come from the 701501–799999 range; legacy data uses 100000–701500.
Technical
generateNextMecaId() queries all profiles with non-null meca_id, filters to the new-system range, finds the max, and returns max+1 as a string. Auto-assigned on profile creation if none provided.
Why
Numeric MECA IDs are the human-friendly primary key members memorize. Splitting old vs new ranges keeps the numbering monotonic for the new system without colliding with imported legacy IDs. Modification is restricted to two super-admin emails because changing a MECA ID retroactively breaks results history.
Manage
Read-only for normal admins. See the MECA ID System reference.

Login Access Control (Ban / Maintenance Bypass)

Overview
Per-member flags that override the default access policy: ban prevents sign-in entirely, maintenance bypass grants access during maintenance mode.
Technical
Two boolean columns on profiles: login_banned and maintenance_login_allowed. Ban also sets Supabase ban_duration to ~100 years so existing sessions are kicked. MaintenanceModeGuard reads the flags on every authenticated request — banned users get 403 always; non-admins are blocked when maintenance mode is on unless they have the bypass flag.
Why
Multi-layer enforcement (DB flag + Supabase ban + guard rejection) ensures a banned user is locked out within ~1 minute even if they have a valid access token. Maintenance bypass exists so beta testers and contractors can keep working during a freeze without being elevated to admin.
Manage
/admin/members → bulk action bar. See the Login Access Control deep guide.

Maintenance Mode

Overview
Site-wide kill switch that blocks non-admin access while the team ships changes.
Technical
Single row in site_settings with key maintenance_mode_enabled. MaintenanceModeGuard reads it (cached 30s) on every request. When on: public routes still resolve, admins pass through, members with bypass pass through, everyone else gets a 503 with a "try again later" message. The orange banner is rendered client-side based on the same flag.
Why
A binary site-wide flag is the simplest interlock. Caching for 30 seconds keeps DB load near zero — toggle latency is acceptable. Admin pass-through is required so we can keep working during the freeze.
Manage
/admin/site-settings → toggle. See the Maintenance Mode guide.

Audit Logs

Overview
Every privileged action is captured: who did what, when, against which target, with old/new values.
Technical
admin_audit_log table with rows: admin_user_id, action (snake_case verb), resource_type, resource_id, description, old_values JSONB, new_values JSONB, ip_address, created_at. Service writes from every privileged path (membership_cancel, login_ban, role_change, refund_issued, etc.).
Why
Owners need an authoritative answer to "who did this and why" for compliance, customer disputes, and incident response. Capturing old_values + new_values lets you reconstruct history if a member claims their account was changed unexpectedly.
Manage
/admin/audit-log — filter and search. Audit Log guide.

Login Audit

Overview
Sign-in/sign-out events with IP and user agent for account-takeover investigation.
Technical
Separate from admin audit — this captures every authentication event including failures. Powered by Supabase auth event hooks plus our own session-lifecycle tracking.
Why
Admin audit answers "what did this admin change?" but takeover investigations need "where did this user log in from?" — two separate concerns deserve two separate tables.
Manage
/admin/login-audit — filter by user, IP, date. Login Audit guide.
