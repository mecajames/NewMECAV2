Business Logic: Events

Event creation, registration, judging, check-in, scoring, and post-event flows.

Event Creation

Overview
An event has a date, venue, format(s), classes, an Event Director, and a Judge panel. Members register, attend, and earn points.
Technical
events table with the core fields plus FKs to season and event_director. Judge panel is a join table (event_judge_assignments). Registration window is two timestamps; events open for registration when the window starts.
Why
Modeling events as first-class entities (not just calendar entries) lets us join on them everywhere — results, registrations, points, ratings. Event Director is a single FK because there's exactly one ED per event; the judge panel is many-to-many because events have multiple judges.

/admin/events. Event Management guide.

Event Registration

Overview
Members register for an event, pick their classes, pay, and receive a confirmation email.
Technical
event_registrations row per registration. Member can register for multiple classes in one transaction. Payment goes through Stripe/PayPal; on capture the registration moves to PAID and the confirmation email fires.
Why
Class selection is per registration (not per member) because someone might compete in different classes at different events. Payment-then-PAID is the same atomic pattern as memberships.

QR Check-in

Overview
Staff scans a competitor's digital card (or enters MECA ID manually) to mark them present at the event.
Technical
Member's digital card encodes /card/<membership-id>. Scanner or manual-entry UI looks up by id, validates the registration is for this event, sets checked_in_at on the registration. Idempotent — a second scan is a no-op.
Why
QR is fast at the event front desk. Manual entry handles "phone died" or "left it in the car." Idempotency prevents accidental double-checks-in if the scanner glitches.

Score Sheets

Overview
Templates that drive the per-class scoring UI Judges fill in during competition.
Technical
Score sheet templates are tied to (format, class) pairs. Each defines sections, fields, point allocations. Judges submit scores; submissions are stored with timestamp and judge id.
Why
Score sheets vary by format (SPL is simple dB readings; SQL has multi-section subjective scoring). Templating means we change the scoring rubric without redeploying.

Results Entry (Manual / Excel / TermLab)

Overview
Three ingestion methods for competition results, all sharing the same downstream business logic.
Technical
Manual entry: a form per row with live MECA ID lookup. Excel import: parse the MECA template .xlsx, validate, batch insert. TermLab import: parse the .lab export from TermLab software, map classes through the class-mappings table. All three call the same validation pipeline (member lookup, membership eligibility, class match, points calculation).
Why
ED at the event types in manually. Big-event organizers paste from a shared sheet. SPL specialists prefer TermLab. Three input forms, one business logic — write once, run from three surfaces.

/admin/events → results. Results Entry System + Results Entry Business Logic.

Points Calculation

Overview
Each event placement earns points; points roll up into season standings and leaderboards.
Technical
Points come from points_configurations indexed by (place, class, format). On result insert, the system looks up the row and assigns it to the result. Standings are recomputed nightly (or on-demand from admin).
Why
Points-per-place is configuration not code so MECA can rebalance scoring between seasons. Per-class config supports different point structures for different competition difficulties.
Manage
/admin/points-configuration. Points Configuration.

Standings & Leaderboard

Overview
Aggregated season-long rankings derived from event results.
Technical
Standings tables are read-only views computed from results. Member standings rank by total points within the season; team standings aggregate per-team. Top-10 and Top-N pages are different cuts of the same underlying data.
Why
Recomputing on every page render would be expensive at scale. Pre-aggregating into materialized views makes the public pages fast and consistent.
Manage
Public-facing — no admin UI other than result corrections that propagate.

Event Hosting Requests

Overview
Members or venues submit "I want to host an event" forms; admin reviews and converts approved ones into actual events.
Technical
Separate event_hosting_requests table with submission fields. Admin review converts approved request to a real event (one-click conversion in the admin UI).
Why
Hosting requests are pre-events — they have venue/date proposals but aren't scheduled yet. Keeping them separate from events keeps the events table clean and avoids "DRAFT events" cluttering the public events list.
Manage
/admin/hosting-requests.

Post-event Rating Requests

Overview
Members are emailed the day after an event they attended, asking them to rate the Judges and ED.
Technical
Daily cron iterates events that ended yesterday; for each registered competitor, send the rating-request email with deep links into the rating form. Ratings drive the per-judge / per-ED averages on the Ratings Analytics page.
Why
Ratings collected immediately are higher fidelity than ratings collected weeks later. Automating the request via cron means it never gets forgotten.
