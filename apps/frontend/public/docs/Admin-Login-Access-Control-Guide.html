Login Access Control — Ban, Unban, Maintenance Bypass

Multi-system operation: DB flag, Supabase ban_duration, audit log

/admin/members (bulk action bar)

What banning actually does

Banning a member is a multi-system operation: it sets a flag in our DB, sets ban_duration on Supabase Auth (~100 years), and the API guard rejects banned users on every subsequent request. Active sessions are kicked within ~1 minute, future sign-ins fail at the auth layer. Use this for TOS violations, fraud, or harassment — not for "I want to stop emailing this member."

1. What happens when you click Ban

Three layers updated

LayerChangeEffect
profiles table (DB)login_banned = true, login_banned_at, login_banned_by (admin id), login_banned_reasonMarks the member banned in our system; the maintenance/access guard reads this on every authed request.
Supabase Authban_duration = '876000h' (~100 years)Supabase rejects sign-in attempts and invalidates active sessions on next refresh.
Audit logEntry: login_banCaptures admin user, target, reason, timestamp.

Timeline of effect

  • Within seconds — DB flag set; new requests from the banned user get rejected with 403.
  • Within ~1 minute — banned user's active session is kicked when their access token tries to refresh.
  • Forever — until you unban, sign-in attempts return "User has been banned" from Supabase.

2. Self-protection rules

  • You cannot ban your own account. The system rejects the action.
  • You cannot ban a hard-coded super-admin email (the original platform owners). The system rejects with a clear error.
  • Reason is required and must be at least 5 characters. The reason is saved on the profile row and shown in the row tooltip.

3. Ban click-through

1
Find the member
Open /admin/members. Search by name, email, or MECA ID.
2
Tick their row
Use the checkbox on the left. The bulk action bar appears at the top. You can tick multiple rows to ban a group at once.
3
Click "Ban Login"
A reason prompt appears. Enter at least 5 characters describing why. Press OK.
4
Confirm
A confirmation dialog warns that active sessions will be kicked. Confirm to proceed.
5
Verify
The row updates to show a red Banned badge with the reason in the tooltip. The member is logged out within ~1 minute. Audit log shows the entry.

4. Unbanning

The Unban Login button on the bulk action bar only appears when at least one selected row is currently banned. Clicking it:

  • Clears login_banned and the audit fields on the profile.
  • Calls Supabase to set ban_duration: 'none' (lifts the auth-layer ban).
  • Writes login_unban to the admin audit log.
  • The user can sign in again on their next attempt. The Banned badge disappears from the row.

5. Audit trail

Open /admin/audit-log and filter by action:

  • login_ban — admin banned a member.
  • login_unban — ban was lifted.
  • login_allow_maintenance — admin granted maintenance bypass to a member.
  • login_revoke_maintenance — bypass revoked.

Each entry includes the admin user, target member, reason (when provided), and timestamp. Use this for compliance review and incident response.
