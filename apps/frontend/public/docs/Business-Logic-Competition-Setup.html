Business Logic: Competition Setup

The configuration data that drives every event: classes, formats, seasons, rulebooks, points, and how legacy/imported data normalizes against current entities.

Formats

Overview
Top-level scoring categories: SPL, SQL, Show & Shine, Dueling Demos.
Technical
competition_formats table — display name, scoring style, active flag.
Why
Formats are the broadest grouping. Putting them in their own table (rather than enum) means we can launch new formats (e.g. RTA, Bass Race) without a deploy.
Manage
/admin/formats. Format Management.

Classes

Overview
Specific competition divisions within a format and season (e.g. SPL Stock Class 1, SQL Pro).
Technical
competition_classes with FK to format and season, plus is_active and unlimited_wattage flags.
Why
Classes are scoped to (format, season) because the same name can mean different things across years. Inactive flag (vs delete) preserves historical results.
Manage
/admin/classes. Classes Management.

Seasons

Overview
Year-scoped containers that own classes, points, events, and standings.
Technical
seasons with start/end dates and is_current/is_next flags. Only one current at a time. Switching current cascades nothing — historical events keep their season.
Why
Most platforms model years as date filtering; we use first-class season entities because MECA reuses class names and points configs across seasons. Having a season FK on every event is unambiguous.
Manage
/admin/seasons. Season Management.

Rulebooks

Overview
PDF rulebooks per (format, season) — the source of truth for what counts as fair competition.
Technical
rulebooks table with FKs to format and season; the PDF lives in object storage. Public Rulebooks page lists active rulebooks; archived rulebooks live in the Archive.
Why
Rulebooks change between seasons; archiving (vs deleting) preserves the historical rulebook a competitor was scored against.
Manage
/admin/rulebooks. Rulebook Management.

Points Configuration

Overview
Tabular config of how many points each placement earns per class.
Technical
points_configurations indexed by (place, class, format). Edited via the admin UI; result inserts look up against this table at the time of the insert.
Why
Locking points at insert-time means if the config changes mid-season, already-entered results are stable. New results going forward use the new config.
Manage
/admin/points-configuration. Points Configuration.

Class Name Mappings

Overview
Map historical or imported class names to current canonical classes.
Technical
class_name_mappings table — source name → canonical class id. Excel/TermLab importers consult this table when their input class doesn't match a current class directly.
Why
Old data uses old names; renaming the canonical class to match would break current registrations. A separate mapping table lets both coexist.
Manage
/admin/class-mappings. Class Mappings.

Achievements / dB Clubs

Overview
Automatic badges awarded when a member meets a numeric threshold (e.g. "140 dB Club" for hitting 140 dB at any sanctioned event).
Technical
Achievement definitions live in achievements. After result insert, a hook checks every applicable achievement and grants achievement_recipients rows for any newly-met threshold. Member profile and dashboard surface earned badges.
Why
Hard-coding achievement rules in code would make adding new ones a deploy. Configurable thresholds let MECA introduce a 145 dB Club, a SQL Champions Club, etc. without engineering work.
Manage
/admin/achievements. Achievements guide.
