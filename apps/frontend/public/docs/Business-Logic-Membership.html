Business Logic: Membership System

How memberships are sold, renewed, paused, cancelled, refunded, and recovered after failed payments.

Membership Types Configuration

Overview
Define the products members can buy: Individual, Family/Secondary, Team add-on, Forever, retailer/manufacturer tiers.
Technical
Each type lives as a row in membership_type_configs with category (competitor / retail / manufacturer / team), price, term, included features, and an is_active flag. Public Membership page renders only active types; existing memberships keep working even if their type is later inactivated.
Why
Types are configuration not code so we can launch new tiers without a deploy. Inactive vs deleted matters because deleting would orphan historical records.
Manage
/admin/membership-types. Membership Types guide.

Membership Purchase Flow

Overview
Member picks a type, completes checkout, payment captures via Stripe or PayPal, membership activates instantly.
Technical
Frontend Stripe Elements (or PayPal SDK) tokenizes payment. Backend creates membership row in PENDING, calls Stripe/PayPal to capture, on success marks PAID and triggers welcome email + admin notification. MECA ID is generated if the user didn't have one.
Why
Tokenized payment means the platform never touches raw card data — PCI compliance is Stripe's problem. PENDING → PAID atomic transition prevents a half-charged-half-not-active state.
Manage
Self-service for members. Admin can also create-and-charge via member detail. Cancel/Refund/etc deep guide.

Auto-Renewal (Stripe Subscriptions)

Overview
Recurring memberships renew automatically via Stripe at term end.
Technical
Membership has stripeSubscriptionId. Stripe fires invoice.payment_succeeded webhook which extends the membership end date and sends the Renewal Confirmation email. Failures fire the dunning flow.
Why
Stripe handles the hard parts (retry logic, network failures, dunning emails on their side). The platform just listens for events and updates state. PMPro-style "legacy" recurring is preserved as a status badge so we can spot members who need to re-set-up auto-renew.
Manage
Member self-service via Stripe Billing Portal. Admin can pause via member detail. Pause section.

Cancel — Now vs At Renewal

Overview
Two cancel paths with different effects on access and refunds.
Technical
cancelMembershipImmediately sets paymentStatus=CANCELLED, ends access right away, no refund. cancelMembershipAtRenewal sets cancelAtPeriodEnd=true, keeps access until endDate, prevents auto-renew. Both write to audit log; cancel-now also fires the admin notification.
Why
Members typically want "stop renewing but let me finish my year" (cancel-at-renewal). Admins enforcing a TOS violation want "end access right now" (cancel-now). One verb wouldn't cover both — separating them makes the intent unambiguous.
Manage
Member detail page or bulk action bar on /admin/members. Deep guide.

Refunds (Full and Partial)

Overview
Issue a Stripe refund, optionally cancel the membership, send a notification email.
Technical
Full refund: Stripe refund for full amount, paymentStatus=REFUNDED, cancelledAt=now, refund email sent. Partial refund: Stripe refund for specified cents, membership stays ACTIVE, note appended to cancellationReason. Stripe failure rolls everything back so we don't mark something refunded that wasn't actually credited.
Why
Partial refunds preserve the membership because we want goodwill credits ("waive this month") to be reversible. Email ALWAYS fires for refunds because the financial impact deserves explicit communication.
Manage
Member detail or bulk Refund all. Refund section.

Pause & Resume

Overview
Temporarily stop charging without ending the membership.
Technical
Calls Stripe pauseSubscription — no charge at next cycle. Membership row is unchanged in our DB. Resume calls Stripe to lift the pause. Audit log: membership_pause / membership_resume.
Why
Pause vs cancel-at-renewal: pause is short-term ("3-week vacation") with no end date implied; cancel-at-renewal commits to ending. Stripe handles the actual not-charging logic so we don't need our own scheduler.
Manage
Member detail page. Pause section.

Dunning (Failed Payment Recovery)

Overview
Escalating reminder emails over 14 days after a renewal payment fails, ending in auto-suspension.
Technical
Daily cron checks memberships with last_dunning_step < max and last_dunning_at < today. Sends step 1 (day 1) → step 2 (day 3) → step 3 (day 7) → step 4 = auto-suspend (day 14). Each step updates last_dunning_step + last_dunning_at so the cron knows where it left off.
Why
Without persisting the step, every cron run would re-send step 1. Without persisting the timestamp, a single run could fire all four steps. Two columns is the minimum to make escalation idempotent and resumable.
Manage
/admin/billing — dunning queue surfaces affected members. Dunning guide.

Secondary Memberships

Overview
Sub-accounts under a primary member (spouse, child) sharing one membership relationship.
Technical
Profiles have master_profile ManyToOne pointing to the primary. is_secondary_account = true and can_login typically false (secondaries usually can't log in independently). Secondaries get their own MECA ID for results tracking but billing flows through the master.
Why
Families and competition couples want one purchase covering everyone. Modeling each secondary as a real profile (rather than a JSON field on the master) lets each one earn results and have a public profile.
Manage
Master member detail → Secondaries tab. Secondary Memberships guide.

Teams (separate from secondaries)

Overview
A competition affiliation. Multiple primary members purchase the Team add-on and elect to join the same Team for standings.
Technical
teams table separate from profiles. Members with the Team add-on can be added to a team's roster. Team standings aggregate the team members' event placements per season.
Why
Teams and secondaries solve different problems. A team is a competition group of independent members; a secondary is a sub-account of one. Conflating them would make team-add billing impossible.
Manage
Admin → Team detail. Team Management guide.

Membership Cards (printed)

Overview
Track the physical-card workflow: requested → printing → shipped → delivered.
Technical
Separate membership_cards tracking table referencing the membership.
Why
Most members are happy with the digital card (QR-coded, printable from the dashboard). For members who request a physical card we need an explicit print/ship workflow because that involves real-world logistics.
Manage
/admin/membership-cards. Membership Cards guide.
