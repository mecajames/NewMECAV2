Send Notifications & Broadcasts

In-app notifications to single members, audience-targeted broadcasts, and admin-side alerts

/admin/notifications · /admin/members (per-row Send Message)

Two ways admins reach members

In-app notifications show in the bell icon on the member's nav bar. Email goes to their inbox. Most admin-driven communication is in-app; email is reserved for transactional events (purchase receipt, password reset, etc.) and the weekly digest. This guide covers in-app notifications. Email is covered separately in the Email Reference Matrix.

1. Notification types

TypeColorUse for
messageDefault / blueDirect human-to-human messages from admin to member
systemGrayAutomated platform messages (e.g. "your event registration was confirmed")
alertRedTime-sensitive, action-required (e.g. "your payment failed")
infoSky blueInformational broadcasts (e.g. "World Finals registration is now open")

Each notification has a title, a markdown-supported body, and an optional link that deep-links to a page when the user clicks the notification.

2. Send to a single member

This is the everyday path — quick replies, requests for clarification, applauding a result, etc.

1
Open /admin/members
Find the recipient.
2
Click "Send Message" on their row
A modal opens with subject and body fields.
3
Compose
Subject is short. Body supports Markdown — use **bold**, *italic*, links, and lists. The message lands in the recipient's notification bell within seconds.
4
Send
The notification is created with type = message, fromUserId = your admin id, and link = "dashboard" by default. The recipient sees it next time they load any page.

3. Broadcast to all active members or all users

For announcements that touch a wide audience (event reminders, season opening, policy changes).

Audience options

  • All active members — everyone with membership_status = active at the moment of send. ~3000 members at peak season; the system writes one notification row per recipient.
  • All users — every profile, including expired members and basic accounts. Use sparingly.
  • Specific user IDs — pass a list (e.g. all judges, all retailers in a region). The admin UI surfaces a filterable picker.

How to send a broadcast

1
Open the broadcast composer
From /admin/notifications click "+ Broadcast" (or use the equivalent in your admin layout).
2
Pick audience
All active / all users / specific IDs. The composer shows a count next to each option (e.g. "All active members (2,847)").
3
Set type, title, body, link
Choose alert/info/message/system. Title under 60 chars. Body up to 1000 chars; Markdown supported. Link is optional but recommended (e.g. /events/abc-123).
4
Confirm — the count is shown again
The system writes one notification row per recipient. For ~3000 active members the broadcast completes in a few seconds.
5
Verify
Look up your own account in /admin/members; you should see the notification yourself if you were in-audience. The Notifications Center will show the broadcast.

Broadcasts are not undoable

Once written, you can delete individual notifications via the admin tool but you cannot un-send. Proofread the title and body before confirming.

4. Notification analytics & the admin notifications center

Open /admin/notifications to see:

  • Total notifications sent in the last 7 / 30 / 90 days.
  • Read rate — what percent of recipients opened them.
  • By type breakdown — which categories drive engagement.
  • Recent broadcasts list with click-through to view the body and recipients.

Each notification row in the admin view has a delete button — useful to clean up a typo'd broadcast or take down a stale message.
