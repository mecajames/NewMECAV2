Class Mappings

Normalize imported class names AND re-link results whose class disappeared

Admin Dashboard → Competition Setup → Class Mappings

Audience: site administrators

The Class Mappings page is the single place to fix every "this result doesn't have a real class" case in the system. It covers two distinct problems that look similar from a user's perspective (results don't show up on the public site) but have different root causes and different fixes:

  • Unmapped Classes Found — a CSV import contained a class name the system has no class_name_mapping rule for. The result was saved with class_id = NULL.
  • Results With Broken Class Link — a result has a class_id, but the class it points to has since been deleted or set inactive (and no other active class shares its name/format).

Both cases hide results from the public Results page, leaderboards, and standings until you resolve them. The page surfaces both at the top so nothing slips through unnoticed.

1. Concepts: the three ways a result identifies its class

Every row in the competition_results table carries three pieces of class information. The public Results page tries each in turn:

Field What it is How it gets set
class_id Foreign key to the competition_classes table. Authoritative. Manual entry: picked from a dropdown. CSV import: resolved via a class_name_mappings rule or by exact name match.
competition_class Free-text class name on the result row itself. Fallback if class_id doesn't resolve. Copied from CSV import. Auto-synced from class_id when the result is created or edited.
format Free-text format (SPL, SQL, Show & Shine, etc). Used alongside competition_class for text fallback. Copied from CSV import. Auto-synced from class_id when the result is created or edited.

The public lookup logic:

  1. If class_id resolves to an active class — use it.
  2. Otherwise, try a case-insensitive match of competition_class against the name/abbreviation of any active class with the same format — use it if found.
  3. Otherwise, the row is hidden from public display and shows up on this page as either Unmapped or Orphan.

Why two separate sections?

A row with class_id = NULL is fundamentally different from a row with class_id = 'abc-123' that points to a deleted class. The first one needs a mapping rule so future imports don't have the same problem. The second one needs to be repointed to the new class. The page treats them differently because the fix is different.

2. Unmapped Classes Found (class_id is NULL)

The yellow alert at the top lists every class name that appeared in an import but has no class_name_mappings rule pointing it at a canonical class. The rows behind those names were saved with class_id = NULL and are hidden from public results.

1
Click the eye icon on an unmapped class card.
A modal opens showing every result row that came in with that exact class name. Verify the rows look like what you expect — same scoring style, similar competitor list, the format matches.
2
In the modal, pick a target class from the dropdown.
You can check individual rows to remap only a subset, or leave none checked to remap all of them.
3
Click Remap Results.
This updates each row's class_id and rewrites the competition_class text to the target class's name. The rows immediately appear on public results.
4
Optional: also create a permanent mapping rule.
Click the + (plus) icon on the unmapped card instead of the eye. This opens the Add Mapping form pre-filled with the source name. Saving the mapping means future imports of the same source name will be auto-routed — you won't see this card again next time.

Best practice: do both

For most unmapped classes you'll want to both remap the existing rows (eye → pick class → Remap) and create a mapping rule (+ → fill form → Save). The first fixes history; the second prevents the same problem on the next import.

3. Results With Broken Class Link (class_id points at a dead class)

The red alert directly below the yellow one lists groups of results whose class_id points to a class that's been deleted or set inactive, AND no active class with the same name/format exists for text fallback to catch.

Common causes:

  • An admin deleted a class that still had results attached to it.
  • A class was renamed (so the old name no longer matches anything active).
  • A season-rollover script created new class IDs for the new season but old rows still point to the previous season's class IDs.
1
Read the diagnostic chips under each group.
They tell you exactly why this group is orphaned and what the recommended fix is. See section 4.
2
Pick a target class in the searchable dropdown on the right of the group header.
Type any substring of the format, class name, or abbreviation. Multi-token search works — typing spl ext matches "SPL Extreme" even when those words aren't adjacent.
3
Click the orange Link N button.
Confirm the prompt and every row in the group is repointed to the chosen class. The competition_class text and format are auto-synced from the new class.
4
Or: expand the group with the chevron and pick specific rows.
Useful when one group accidentally contains results that should be split across two different classes. Use the per-row checkboxes plus the sticky bottom toolbar to bulk-link selected rows — see section 5.

4. Reading the diagnostic chips

Each orphan group shows up to three coloured chips that explain why it's broken and what the system thinks the fix is.

Chip Meaning Action
class_id → X (INACTIVE / DELETED) The class_id on these rows points to a class that exists in the DB but is no longer usable. Often the original class was renamed or replaced; the row text (e.g. "DDM") might no longer match the linked class name (e.g. "Park and Pound") because they were synced at different times. Pick the current correct class in the dropdown and click Link.
class_id → none (NULL) Edge case where the row also has no class_id and isn't covered by the Unmapped section (usually because the competition_class text is empty too). Pick a target class. If the rows are genuinely unidentifiable, you may want to delete them through Results entry.
CSV mapping "X" → Y There IS an active mapping rule for this exact source text in the class_name_mappings table. It points at the class named "Y". This is your most likely candidate — the rule wasn't applied during import (or was added after these rows were imported). Pick that target class in the dropdown — it's pre-suggested when you open the picker.
name match: X An active class with the same name or abbreviation exists somewhere in the system, but didn't catch the text fallback because the format on the row doesn't match the class's format. Could be a typo on the format field, or a class that should accept this format. Use only if you're confident the suggested class is correct. Check the format mismatch first.

5. Bulk-selecting rows across groups

When you need to move rows from several groups into the same target class (e.g. all "X" rows whether they're tagged SPL or SQL), use the per-row checkboxes instead of the per-group Link button:

  1. Expand any group with the chevron at the left of its header.
  2. Check individual rows or use the group's "select all" checkbox at the top of the row table.
  3. Repeat for other groups — selections persist when you collapse and expand groups.
  4. A sticky toolbar appears at the bottom of the screen showing the running count.
  5. Pick a target class in the toolbar's searchable dropdown and click Link N.

The toolbar's Clear button drops the selection without linking anything.

6. Managing the mappings list

The bottom section of the page lists every saved mapping rule. These are the rules that get consulted on every CSV import.

1
Add a mapping
Click + Add Mapping in the page header. Enter the source name exactly as it appears in import files, pick the target canonical class, and Save. The next import that contains that source name will route automatically.
2
Edit a mapping
Click the yellow pencil icon on the row. Adjust source name or target class as needed. Editing a mapping affects future imports only — already-imported rows keep their existing class_id.
3
Deactivate (don't always delete)
Set a mapping to Inactive instead of deleting it when you want to stop matching it without losing the history. Inactive mappings don't apply to imports but remain visible in the list.
4
Search and filter
Use the search box for source name or target class text. The status filter narrows to All / Active / Inactive.

7. Troubleshooting

"I linked the rows but they still don't show on public results."

The public Results page caches results data for a few minutes. Click Refresh on the public page, or wait 5 minutes for the cache to expire. If still missing, double-check that the target class is marked Active on the Classes Management page.

"The group header says class_id points to 'Park and Pound' but the row text says 'DDM'."

That's exactly the symptom this page is designed for. The row was imported with a CSV mapping that pointed "DDM" → "Park and Pound", so the class_id was set correctly at import time. Then the target class was deleted or made inactive. The row text wasn't updated when the class was deleted, so you see the old DDM text plus a stale class_id.

"I see the same source name in BOTH Unmapped Classes and Results With Broken Class Link."

This means some rows for that name have class_id = NULL (Unmapped) and others have a dangling class_id (Orphan). Handle them as two separate fixes: remap the NULL ones via the Unmapped modal, and link the orphan ones via the orphan group's Link button. Both should resolve to the same target class.

"My class is missing from the dropdown."

The dropdown only shows classes that exist in the system. If you need a class that isn't there yet, open Classes Management first to create it, then come back and refresh this page.
