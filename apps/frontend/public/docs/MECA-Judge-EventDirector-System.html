MECA Judge & Event Director System

Implementation Plan Overview

Implementation Phases

  • Phase 1: Database Foundation
  • Phase 2: Judge Application System
  • Phase 3: Admin Judge Management
  • Phase 4: Event Director System
  • Phase 5: Dashboard Integration
  • Phase 6: Seasonal Qualification
  • Phase 7: Event Integration
  • Phase 8: Public Directories
  • Phase 9: Rating System
  • Phase 10: Enhanced Features

Phase 1: Database Foundation & Core Infrastructure

Goal: Create all database tables, migrations, and shared utilities

Backend Tasks

  • Add judge role to UserRole enum
  • Create database migrations for judge/ED tables
  • Create MikroORM entities for all new tables
  • Create shared Zod schemas for DTOs

Deliverable: All database tables created and entities defined

Phase 2: Judge Application System (Public-Facing)

Goal: Allow members to apply to become judges

Key Features

  • Public information page: /judges/become-a-judge
  • Email verification for non-logged-in users
  • Multi-step application form with sections for personal info, headshot, location, experience, references
  • Application status tracking

Deliverable: Members can visit info page and apply to become judges

Phase 3: Admin - Judge Application Management

Goal: Admins can review, approve, and reject judge applications

Key Features

  • Admin applications list page with filters
  • Application detail page with reference check tracking
  • Admin ability to create applications on behalf of members
  • Approved judges list page

Deliverable: Admins can view, review, approve/reject applications

Phase 4: Event Director System

Goal: Parallel system for Event Directors

Key Features

  • ED application system (mirrors judge system)
  • ED-specific fields (event management experience, team experience)
  • Admin ED management pages

Deliverable: Full ED application and admin management system

Phase 5: Judge & ED Dashboard Integration

Goal: Approved Judges/EDs can manage their profiles from dashboard

Key Features

  • Conditional "Judge Profile" tab for approved judges
  • Conditional "Event Director Profile" tab for approved EDs
  • Headshot management, bio editor, availability settings

Deliverable: Users can manage their Judge/ED profiles

Phase 6: Seasonal Qualification Management

Goal: Track and manage seasonal qualifications

Key Features

  • Season qualifications section in admin detail pages
  • Bulk qualify judges for season
  • Season status filtering

Deliverable: Admins can qualify Judges/EDs for seasons

Phase 7: Event Integration & Judge Assignments

Goal: Assign judges to events, update ED assignment

Key Features

  • Event Director dropdown shows only approved EDs
  • Judge assignments section on event detail page
  • Judge "Available Events" view in dashboard
  • Volunteer/request system for judges

Deliverable: Admins can assign judges to events, judges can volunteer

Phase 8: Public Directories

Goal: Public-facing directories for Judges and Event Directors

Key Features

  • Public judges directory: /judges
  • Public ED directory: /event-directors
  • Individual profile pages with headshot, bio, statistics

Deliverable: Public can browse judge and ED directories

Phase 9: Rating System

Goal: Allow eligible users to rate Judges and Event Directors

Key Features

  • Rating submission modal (5-star)
  • Eligibility verification (competed at same event)
  • 30-day rating window
  • Admin ability to remove fraudulent ratings

Deliverable: Users can rate judges/EDs they've interacted with

Phase 10: Enhanced Features & Polish

Goal: Level progression, availability management, statistics

Key Features

  • Judge level change tracking with history
  • Availability calendar in dashboard
  • Bulk operations (level change, export, status change)
  • Email notification preferences

Deliverable: Complete level progression and availability management
